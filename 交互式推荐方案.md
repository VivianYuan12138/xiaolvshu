# 小绿书 — 交互式推荐技术方案

> 核心问题：如何让系统"越用越聪明"，同时保证内容质量？

## 1. 现状与问题

### 当前架构（开环）

```
RSS 信息源(18个) → 抓取 → AI评分(固定Prompt) → AI改写 → 展示
                                                          ↓
                              无任何反馈 ← ← ← ← ← ← ← 用户
```

- 评分 Prompt 是静态的，不会因用户行为而改变
- 评分维度偏"信息密度"，导致数据堆砌的财经文章霸榜，用户觉得"无聊"
- 用户收藏、阅读时长、跳过等行为信号完全未被利用
- 没有 exploration 机制，容易困在固定内容类型里

### 目标架构（闭环）

```
RSS → [质量门槛(不可变)] → [自适应评分(可进化)] → [探索/利用平衡] → 展示
                                    ↑                                  ↓
                          偏好画像 + 反思 + 规则库              用户行为信号
                                    ↑                                  ↓
                                    └──────── 反馈闭环 ←──────────────┘
```

### 关键设计原则

| 维度 | 说明 |
|------|------|
| **Quality Score（质量分）** | 客观评估，不随用户行为变化。反焦虑/反标题党是硬约束。 |
| **Relevance Score（相关分）** | 主观匹配，随行为自适应。基于用户偏好动态调整。 |
| **最终排序** | `Quality >= 5` 才通过门槛，然后按 `Quality × Relevance` 排序 |

**Quality 是护栏，不能弯。Relevance 是方向盘，可以转。**

## 2. 竞品调研

### 现有 RSS + AI 项目

| 项目 | 技术 | 自适应学习？ |
|------|------|-------------|
| [auto-news](https://github.com/finaldie/auto-news) | LangChain + LLM 过滤 | ❌ 静态兴趣过滤 |
| [curator-ai](https://github.com/marmelab/curator-ai) | LLM 0-10 评分 | ❌ 手动指定兴趣 |
| [RSSbrew](https://github.com/yinan-c/RSSbrew) | 正则/布尔过滤 + OpenAI 摘要 | ❌ 规则过滤 |
| [Folo](https://github.com/RSSNext/Folo) | AI 翻译 + 摘要 | ❌ 无个性化 |
| [Gorse](https://github.com/gorse-io/gorse) | Go 推荐引擎 | ✅ 协同过滤（但需多用户） |

**结论：目前没有任何开源 RSS+AI 项目实现了单用户自适应学习。这是差异化机会。**

### 为什么传统 RL 不适用

| 问题 | 说明 |
|------|------|
| 数据量不足 | 单用户每天最多 30 次交互，RL 收敛需要万级 |
| Exploration 代价高 | RL 必须推"可能不好的"来试错，违背反焦虑理念 |
| Action space 是 prompt | 决策通过 LLM prompt 做出，不适合梯度优化 |

## 3. 推荐技术方案

### 3.1 CIPHER — 从行为推断偏好原因

**来源**: NeurIPS 2024 — [Aligning LLM Agents by Learning Latent Preference from User Edits](https://proceedings.neurips.cc/paper_files/paper/2024/file/f75744612447126da06767daecce1a84-Paper-Conference.pdf)

**核心思想**: 用户收藏了一篇文章，不只记录"收藏"，而是让 LLM 推断**为什么**收藏。

```
输入:
  用户收藏了: "DFlash推测解码：苹果芯片推理加速4.1倍"
  用户跳过了: "长安汽车2025年报：净利润腰斩"

LLM 推断:
  "用户偏好本地AI推理、开源工具类实操内容，对财报分析类兴趣不大"

→ 存入 preference_reasons 表
→ 下次评分时注入 prompt，影响 Relevance Score
```

**适合小绿书的原因**:
- 最少 5-10 次收藏即可启动
- 纯 LLM 调用，无需训练
- 输出可解释——用户能看到"系统为什么推这个给我"
- 每次交互都产出一条偏好理由，天然适合交互式推荐

**改进版 PROSE** (Apple, ICML 2025): 迭代式精炼推断结果，比 CIPHER 准确率高 33%。

### 3.2 MemGPT 模式 — 持久偏好记忆

**来源**: [MemGPT / Letta](https://research.memgpt.ai/) — 2024-2025

**核心思想**: 维护一段不断更新的"用户画像文本"，始终注入每次 LLM 调用的 prompt。

```
架构:
  Core Memory（始终在 prompt 中）: 用户偏好摘要（~200字）
  Archival Memory（按需检索）:  完整交互历史（SQLite）

用户画像示例:
  "这位用户是技术背景，关注:
   - AI工具和开源项目（尤其是本地部署相关）
   - 独立开发者/一人公司故事
   - 有趣的技术 hack 和逆向工程
   不感兴趣:
   - 财报分析和宏观经济
   - 纯新闻转述（没有深度分析的）
   - 产品营销软文"
```

**与 CIPHER 的配合**:
- CIPHER 产出单条偏好原因（细粒度）
- MemGPT 将多条原因聚合为画像摘要（全局视图）
- 画像定期更新（每天/每周），注入评分和改写 prompt

**偏好漂移检测** (PAMU, arxiv 2510.09720):
- 对比近期偏好 vs 历史偏好，检测兴趣变化
- 防止画像变"过时"——用户可能从关注 AI 转向关注硬件

### 3.3 Reflexion — 评分自我反思

**来源**: [Reflexion: Language Agents with Verbal Reinforcement Learning](https://arxiv.org/abs/2303.11366) — 2023

**核心思想**: 定期让 LLM 反思自己的评分准不准，产出"verbal feedback"存储并影响后续决策。

```
反思 Prompt:
  "你给这些文章打了高分但用户全部跳过: [列表]
   你给这些打了低分但用户搜索找到并收藏了: [列表]
   你发现了什么规律？下次评分应该调整什么？"

LLM 反思输出:
  "我高估了纯数据堆砌的财经文章——有大量数字不等于有阅读价值。
   我低估了有实操步骤的技术教程——哪怕篇幅短，用户也觉得有用。
   调整建议: 提高'实操性/可复现性'权重，降低'数据密度'的权重。"

→ 存入 reflections 表
→ 最近 3 条反思注入评分 prompt
```

**Voyager 规则库扩展** (ICLR 2025):
- 从反思中提炼可复用的评分规则
- 存入 `scoring_rules` 表，附带置信度
- 示例: `Rule #7: 有完整代码示例的技术文章 → +1.5分 (置信度: 0.85, 基于 12 次交互)`
- 类似 Voyager 的 Skill Library，但用于评分而非游戏

### 3.4 DSPy — 自动优化 Prompt

**来源**: [DSPy](https://github.com/stanfordnlp/dspy) — Stanford NLP, 2024-2025

**核心思想**: 把"prompt 调优"变成自动化的优化问题。用用户收藏作为 ground truth，自动搜索最佳 prompt。

```python
import dspy

# 定义评分模块
class ArticleScorer(dspy.Module):
    def forward(self, title, content):
        return dspy.Predict("title, content -> score, tags, summary")(
            title=title, content=content
        )

# 定义优化目标: 用户收藏的文章应该获得高分
def metric(example, prediction):
    if example.is_favorited:
        return prediction.score >= 7
    return prediction.score < 5

# 自动优化（尝试不同 prompt 措辞 + few-shot 示例）
optimizer = dspy.BootstrapFewShot(metric=metric, max_bootstrapped_demos=4)
optimized_scorer = optimizer.compile(ArticleScorer(), trainset=user_data)
```

**优化器选项**:

| 优化器 | 数据需求 | 说明 |
|--------|---------|------|
| BootstrapFewShot | 10-20 条 | 自动选择最佳 few-shot 示例 |
| COPRO | 50-100 条 | 坐标上升优化 prompt 措辞 |
| MIPROv2 | 100-200 条 | 贝叶斯优化，instruction + few-shot 联合调优 |

**适合小绿书**: 每月积累足够收藏数据后，跑一次优化，自动产出新版评分 prompt。

### 3.5 LLM 初始化的 Contextual Bandit — 探索新兴趣

**来源**: [Jump Starting Bandits with LLM-Generated Prior Knowledge](https://aclanthology.org/2024.emnlp-main.1107/) — EMNLP 2024

**核心思想**: 用 LLM 生成初始偏好先验解决冷启动，然后用 Thompson Sampling 平衡探索与利用。

```
冷启动（第一天）:
  LLM 先验: "技术用户大概率喜欢 AI/开源 → 高置信度
             可能喜欢科学发现 → 中置信度
             不确定生活方式类 → 低置信度"

第 7 天（10 次交互后）:
  用户收藏了一篇"城市漫步指南" → Thompson Sampling 更新
  生活方式类置信度上升 → 开始推更多类似内容

第 30 天:
  真实数据已覆盖大部分类目 → LLM 先验权重自然衰减
```

**解决的核心问题**: 信息茧房
- 有理论保证的 exploration（UCB/Thompson Sampling 的 regret bound）
- 不是随机推垃圾，而是在"不确定"的类目中优先探索
- 后验论文验证 ([arxiv 2604.02527](https://arxiv.org/html/2604.02527v1)): LLM 先验在 30% 噪声内仍有效

**推荐库**: [contextualbandits](https://github.com/david-cortes/contextualbandits) (Python)

### 3.6 Constitutional AI — 质量底线

**来源**: [Anthropic Constitutional AI](https://www.anthropic.com/news/constitutional-classifiers) + [C3AI, ACM WWW 2025](https://dl.acm.org/doi/10.1145/3696410.3714705)

**核心思想**: 定义不可动摇的"内容宪法"，作为质量评估的硬约束。

```markdown
# 小绿书内容宪法

## 必须遵守（违反即低分）
1. 不得贩卖焦虑（"你不知道就完了"、"再不XX就晚了"）
2. 不得标题党（标题与正文严重不符）
3. 不得夹带未标注的广告/软文
4. 不得传播未经验证的信息

## 质量标准（加分项）
1. 有具体数据/案例支撑的分析
2. 有可复现的实操步骤
3. 提供独特视角或一手信息
4. 写作清晰，结构合理

## 不可被用户偏好覆盖
即使用户高频收藏某类内容，以上规则仍然生效。
质量分 < 5 的文章永远不展示。
```

**这是整个系统的"护栏"——不管其他组件怎么进化，宪法不变。**

## 4. 系统架构

### 完整数据流

```
RSS 文章
    │
    ▼
┌───────────────────────────────────┐
│  Constitutional Quality Filter     │  ← 静态规则，始终生效
│  (内容宪法 → 硬约束)                │
└──────────────┬────────────────────┘
               │ Quality >= 5 才通过
               ▼
┌───────────────────────────────────┐
│  自适应评分 Prompt                  │  ← 动态注入以下内容:
│  ┌─ user_preference_summary       │    (MemGPT 用户画像)
│  ├─ recent_reflections            │    (Reflexion 自我反思)
│  ├─ scoring_rules                 │    (Voyager 规则库)
│  ├─ preference_reasons            │    (CIPHER 偏好原因)
│  └─ content_constitution          │    (Constitutional AI 宪法)
└──────────────┬────────────────────┘
               │
               ▼
┌───────────────────────────────────┐
│  Contextual Bandit                 │  ← 探索/利用平衡
│  (偶尔推"不确定"类目的内容来探索)     │
└──────────────┬────────────────────┘
               │
               ▼
           展示给用户
               │
               ▼
┌───────────────────────────────────┐
│  行为信号收集                       │
│  收藏(+3) / 读完(+1) / 秒关(-1)   │
└──────────────┬────────────────────┘
               │
               ▼
┌───────────────────────────────────┐
│  反馈处理层                         │
│  ├─ CIPHER: 推断偏好原因            │  (每次收藏触发)
│  ├─ MemGPT: 更新用户画像            │  (每日聚合)
│  ├─ Reflexion: 反思评分准确度        │  (每周运行)
│  ├─ Voyager: 提炼/更新评分规则       │  (每周运行)
│  └─ DSPy: 优化 Prompt              │  (每月运行)
└───────────────────────────────────┘
```

### 数据库设计

```sql
-- 用户行为信号
ALTER TABLE articles ADD COLUMN opened_at TEXT;          -- 打开时间
ALTER TABLE articles ADD COLUMN read_duration INTEGER;   -- 阅读秒数
ALTER TABLE articles ADD COLUMN is_favorited INTEGER DEFAULT 0;

-- CIPHER 偏好原因
CREATE TABLE preference_reasons (
  id INTEGER PRIMARY KEY,
  reason TEXT NOT NULL,              -- "用户偏好本地AI部署相关的实操内容"
  source_article_ids TEXT,           -- 推断来源的文章ID
  confidence REAL DEFAULT 0.5,       -- 置信度
  created_at TEXT DEFAULT (datetime('now'))
);

-- Reflexion 反思记录
CREATE TABLE reflections (
  id INTEGER PRIMARY KEY,
  content TEXT NOT NULL,             -- 反思内容
  false_positives TEXT,              -- 高分但用户跳过的文章
  false_negatives TEXT,              -- 低分但用户收藏的文章
  created_at TEXT DEFAULT (datetime('now'))
);

-- Voyager 评分规则库
CREATE TABLE scoring_rules (
  id INTEGER PRIMARY KEY,
  rule TEXT NOT NULL,                -- "有完整代码示例的技术文章 → +1.5分"
  confidence REAL DEFAULT 0.5,       -- 置信度 (0-1)
  hit_count INTEGER DEFAULT 0,       -- 命中次数
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

-- MemGPT 用户画像
CREATE TABLE user_profile (
  id INTEGER PRIMARY KEY,
  summary TEXT NOT NULL,             -- 用户偏好摘要 (~200字)
  updated_at TEXT DEFAULT (datetime('now'))
);
```

## 5. 实现路径

### Phase 1: 最小闭环（5 次交互即可用）

```
工作量: ~3 天
技术: CIPHER + MemGPT
效果: 系统开始"认识"用户

实现:
1. 收藏信号从 localStorage → 后端 API → SQLite
2. 每次收藏触发 CIPHER 推断偏好原因
3. 每日聚合为 MemGPT 用户画像
4. 评分 prompt 注入画像 → Relevance Score
```

### Phase 2: 自我纠错（20 次交互）

```
工作量: ~2 天
技术: Reflexion + Voyager 规则库
效果: 系统开始"修正自己的错误"

实现:
1. 每周跑一次 Reflexion（对比评分 vs 用户行为）
2. 从反思中提炼评分规则，存入规则库
3. 评分 prompt 注入最近反思 + top 规则
```

### Phase 3: 自动优化 + 探索（50+ 次交互）

```
工作量: ~5 天
技术: DSPy + Contextual Bandit
效果: Prompt 自动进化 + 主动打破信息茧房

实现:
1. DSPy BootstrapFewShot 优化评分 prompt（每月）
2. Contextual Bandit 控制内容多样性（每次推荐）
3. 引入 Python 微服务（或用 tsx 调 Python 脚本）
```

## 6. 技术可行性评估

| 技术 | 最少数据量 | 实现复杂度 | 需要 GPU？ | 解决"越用越聪明"？ |
|------|-----------|-----------|-----------|-------------------|
| Constitutional AI (宪法) | 0 | 低 | 否 | ❌ 质量底线，不学习 |
| CIPHER (偏好推断) | 5 次收藏 | 低 | 否 | ✅ |
| MemGPT (偏好记忆) | 5 次收藏 | 低 | 否 | ✅ |
| Reflexion (自我反思) | 10-20 篇 | 低 | 否 | ✅ |
| Voyager 规则库 | 20-30 次交互 | 中 | 否 | ✅ |
| DSPy BootstrapFewShot | 20-50 条 | 中 | 否 | ✅ |
| Contextual Bandit (CBLI) | 0 (LLM 先验) | 中 | 否 | ✅ (exploration) |
| DSPy MIPROv2 | 100-200 条 | 中 | 否 | ✅ |
| PAMU 偏好漂移 | 50+ | 中 | 否 | ✅ |
| Ebbinghaus 记忆衰减 | 100+ | 低 | 否 | 部分 |
| 完整 RL (LERL 等) | 1000+ | 很高 | 是 | ⚠️ 杀鸡用牛刀 |
| DPO 微调 | 500+ pairs | 很高 | 是 | ⚠️ 不需要 |

**关键结论: 所有实用方案都是 prompt 工程级别的，不需要 GPU 训练，完全能跑在 SQLite + LLM API 上。**

## 7. 与小红书的本质区别

| | 小红书 | 小绿书 |
|--|--------|--------|
| 优化目标 | 用户停留时长（engagement） | 用户获得感（quality × relevance） |
| 质量定义 | 高互动 = 好内容 | 质量独立评估，不由行为定义 |
| 焦虑内容 | 高互动 → 算法推更多 | 硬规则拦截，不管互动多高 |
| 信息茧房 | 越刷越窄 | Bandit 保证 exploration |
| 可解释性 | 黑盒 | 用户能看到"为什么推这个" |
| 数据需求 | 百万用户协同过滤 | 单用户 5 次收藏即启动 |
